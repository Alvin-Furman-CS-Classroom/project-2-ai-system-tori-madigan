Logic Puzzle Generation and Analysis System
Abstract
This paper presents the design, implementation, and evaluation of a six‑module artificial intelligence system for generating, solving, validating, analyzing, and explaining logic puzzles. The project integrates core concepts from constraint satisfaction, propositional logic, inference, entailment checking, complexity analysis, and knowledge representation into a unified pipeline. The system accepts puzzle parameters, constructs uniquely solvable puzzle instances, translates constraints into propositional formulas, solves the resulting knowledge base using propagation and backtracking, verifies solution correctness, computes difficulty metrics, and produces human‑readable explanations of reasoning steps. Evaluation was conducted through automated unit and integration tests as well as interactive testing via a Streamlit interface. All tests passed at the time of reporting. Results indicate that the system is robust within its intended scope and provides transparent outputs suitable for both machine‑level validation and player‑facing interaction. The project also documents key design evolutions, including strengthened uniqueness enforcement, enhanced difficulty scoring, and a game‑oriented user interface. Overall, the system demonstrates that a modular symbolic architecture can maintain formal rigor while supporting interpretable, user‑facing puzzle experiences.
Introduction
Logic puzzles offer a compelling domain for artificial intelligence because they combine formal constraints, search, inference, and explainability in a manner that is both technically rich and accessible to human reasoning. Rather than treating these AI concepts independently, this project integrates them into a single pipeline in which each module contributes to a coherent reasoning process. The result is a functional puzzle engine that also serves as a structured environment for demonstrating how knowledge representation and reasoning components interact.
A central motivation of the project is to bridge two goals that are often separated in student‑built systems: formal correctness of symbolic reasoning and practical, user‑facing interpretability. Many systems can internally solve puzzles but provide little transparency into their reasoning. Conversely, puzzle applications designed for players often obscure the underlying logic. Our system intentionally supports both aims by producing machine‑verifiable representations and reports while also generating natural‑language clues and explanation artifacts.
The project scope is intentionally bounded. The system focuses on finite, grid‑structured logic puzzles with symbolic entities, attributes, and values. Supported constraint types include equality, inequality, same/different relations, and relative‑position constraints. Inference is propositional and finite‑domain; the system does not incorporate probabilistic reasoning, large‑language‑model inference, or external knowledge retrieval. The user interface is designed for puzzle play and progress feedback rather than multi‑user interaction or persistent accounts.
Within this scope, the system addresses the full lifecycle of puzzle generation, representation, solving, verification, difficulty analysis, and explanation. It does not attempt to guarantee pedagogically optimized clue sets, provide adaptive tutoring, or scale to very large grid sizes where symbolic search becomes computationally expensive. These limitations are deliberate and aligned with course objectives.
System Architecture
The system architecture is a linear but inspectable pipeline with optional user‑interface layers. The pipeline proceeds from Module 1 (Generator) to Module 2 (Logic KB) to Module 3 (Solver + Proof) to Module 4 (Verifier) to Module 5 (Complexity) to Module 6 (Explanation). Each module has explicit input–output contracts and can operate independently or as part of end‑to‑end integration tests. The design emphasizes structured data transformations across representations, including JSON puzzle structures, text‑based knowledge bases, solver outputs, verification reports, and explanation artifacts.

Module Implementation Summary
Module 1: Puzzle Generator (CSP)
Module 1 generates puzzle instances containing entities, attributes, constraints, and a hidden solution. Inputs include grid_size and difficulty, and outputs include a structured puzzle dictionary. The generator uses a solution‑first approach: it constructs a valid assignment and then derives constraints consistent with that assignment. A uniqueness‑enforcement loop validates candidate puzzles using downstream logic and regenerates them if they are not uniquely solvable. Difficulty is scaled by adjusting constraint density. Early versions attempted to enforce uniqueness later in the pipeline, but this caused unstable regenerate‑and‑recheck cycles; moving uniqueness enforcement into Module 1 resolved this issue.
Module 2: Logic Representation (Propositional Logic)
Module 2 converts puzzle constraints into propositional formulas and assembles a structured knowledge base. It uses a canonical proposition format (E*_A*_V*) for consistent parsing and includes explicit domain rules enforcing at‑least‑one and at‑most‑one constraints. Deterministic templates ensure consistent translation across constraint types. The resulting knowledge base is consumed by the solver and verification modules.
Module 3: Puzzle Solving (Inference)
Module 3 solves the knowledge base and produces both a full assignment and a proof trace. The solver applies forward constraint propagation followed by backtracking with MRV‑style variable selection. It also includes a bounded solution counter used for uniqueness checks. Outputs include solution lines, proof steps, and inference‑step counts.
Module 4: Solution Verification (Entailment and Constraint Checking)
Module 4 validates solver outputs against the original constraints and the knowledge base. It produces a structured verification report and a text rendering summarizing pass/fail status, per‑constraint checks, and violation details. The module uses a parse‑and‑check approach for both puzzle constraints and KB‑derived formulas and includes defensive handling for malformed or partial solver outputs.
Module 5: Complexity Analysis
Module 5 computes difficulty metrics and produces interpretable difficulty signals. Metrics include constraint count, search‑space size, inference‑step count, constraint density, formula complexity, branching factor, and uniqueness indicators. The module supports both historical‑baseline scoring and fallback scoring when baseline data is unavailable. Outputs include structured metrics and text reports.
Module 6: Explanation (Knowledge Representation and Natural‑Language Reasoning)
Module 6 translates solver outputs and proof traces into human‑readable explanations. It groups reasoning into phases, extracts deduction steps, and generates natural‑language descriptions. Early versions produced repetitive or overly literal phrasing; later iterations introduced mixed clue types, redundancy pruning, and varied wording to improve readability. The Streamlit UI can display these explanation artifacts.
Evaluation Methodology
Evaluation focused on correctness, robustness, and coherence across modules. Three complementary methods were used: unit tests, integration tests, and interactive UI testing. Tests evaluated generation validity, uniqueness behavior, formula translation consistency, solver correctness, verification accuracy, complexity‑metric completeness, explanation readability, and UI behavior.
Metrics included test pass/fail counts, verification‑report fields, complexity‑metric sanity checks, and UI behavioral checks such as ensuring hidden solutions remained concealed during gameplay. Random‑seeded puzzle instances were tested across difficulty levels and grid sizes. At the time of reporting, all 124 tests in the project passed.

Results
The system met all functional goals and demonstrated reliable end‑to‑end behavior. A key strength is modular transparency: each stage is independently inspectable while remaining compatible with downstream modules. Verification is particularly strong, with clear diagnostics and entailment checks. Difficulty analysis is explicit and interpretable, and explanation generation provides readable reasoning rather than opaque solver outputs. The UI evolved into a player‑oriented interface with clues, guess checking, progressive reveal, and completion feedback.
Limitations remain. Clue quality varies due to template‑driven generation. Search scalability is intentionally bounded to course‑scale puzzles. Difficulty calibration depends on baseline dataset quality.

Proposal Delta
Several changes were made relative to the original proposal. Uniqueness enforcement became stricter, moving from a suggested validation step to a hard requirement. Difficulty scoring expanded to include historical percentile scoring with fallback logic. The UI shifted from developer‑oriented displays to player‑focused gameplay. Module 6 gained refined explanation structure with reasoning‑phase grouping and metadata. No modules were removed; changes focused on robustness and user‑facing quality.
Limitations and Failure Analysis
Natural‑language clue generation can produce repetitive or stylistically uneven output due to template‑driven phrasing. Improvements could include synonym diversification, redundancy penalties, and discourse‑planning techniques.
Symbolic proposition formats constrain naming flexibility; attempts to use real‑world labels can break parsers expecting canonical forms. A mapping layer between display labels and internal IDs would improve robustness.
Difficulty scoring depends on baseline dataset quality; sparse or unrepresentative baselines reduce interpretability. Future work should include curated benchmark sets and confidence diagnostics.

Conclusions and Future Work
The project successfully delivered a coherent six‑module AI system that spans puzzle generation, formal reasoning, verification, complexity analysis, and explanation. The architecture demonstrates that symbolic pipelines can maintain technical rigor while supporting user‑facing interaction. Test results confirm functional stability, and UI evolution shows practical utility.
Future work should focus on improving clue quality, strengthening difficulty calibration through better datasets, and enhancing explanation clarity through discourse planning and visualization. A dedicated language‑realization layer could further improve naturalness while preserving logical precision.
