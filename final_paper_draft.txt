Title and Team Members

Logic Puzzle Generation and Analysis System

This paper was prepared by Tori and Madigan.


Abstract

This paper reports the design, implementation, and evaluation of a six-module AI system for generating, solving, validating, analyzing, and explaining logic puzzles. The project goal is to build an end-to-end pipeline that demonstrates core course topics through a single coherent domain: Constraint Satisfaction Problems (CSP), propositional logic representation, inference, entailment checking, complexity analysis, and knowledge representation. The implemented system accepts puzzle parameters (grid size and difficulty), constructs solvable puzzle instances, converts constraints to logical formulas, solves the puzzle with forward-propagation-plus-backtracking, verifies solution correctness, computes explicit difficulty metrics, and produces a human-readable explanation of solution reasoning. We evaluated the system primarily through automated unit and integration tests across modules and pipeline stages, plus interactive testing via a Streamlit interface. The final codebase includes module-specific tests and full-pipeline integration tests, with all tests passing at the time of this report. Results indicate the system is robust on intended input ranges and provides transparent outputs for both machine-level validation and player-facing interaction. We also document proposal-to-implementation changes, including stronger uniqueness enforcement, historical-baseline difficulty scoring, and a game-oriented UI. The final system demonstrates that a modular architecture can preserve formal rigor while delivering user-facing interpretability.


Introduction

Logic puzzles are a natural domain for AI systems because they combine formal constraints, search, inference, and explainability in a way that is both technically rich and human interpretable. Instead of treating each AI concept in isolation, this project frames multiple topics in a single pipeline where each stage feeds the next. The result is not only a working puzzle engine but also a structured environment for demonstrating how knowledge representation and reasoning components cooperate.

The core motivation of the project is to bridge two needs that are often separated in student projects: formal correctness of symbolic reasoning and practical user-facing outputs. Many systems can solve puzzles internally but do not provide transparent diagnostics or explanations. Conversely, user-friendly puzzle apps often conceal algorithmic machinery. Our system intentionally supports both by producing machine-verifiable representations and reports while also offering natural-language clues and explanation outputs.

Scope is intentionally bounded. The system focuses on finite, grid-structured logic puzzles with symbolic entities, attributes, and values. Constraint types are limited to equality, inequality, same/different value, and relative-position style relations. Inference is propositional and finite-domain, so there is no probabilistic reasoning, large-language-model reasoning, or external knowledge retrieval in core solving. The UI is designed for puzzle play and progress feedback, not for multi-user online gameplay or persistent user accounts.

Within this scope, the system addresses the full lifecycle of generation, representation, solving, verification, difficulty analysis, and explanation. What it does not do is guarantee pedagogically optimal clue sets for every generated puzzle instance, provide adaptive tutoring, or optimize for very large grid sizes where symbolic search costs may grow significantly. These limits are deliberate and aligned with course objectives.


System Architecture

The architecture is a linear but inspectable pipeline with optional UI surfaces, following the flow Module 1 (Generator) to Module 2 (Logic KB) to Module 3 (Solver + Proof) to Module 4 (Verifier) to Module 5 (Complexity) to Module 6 (Explanation). Each module has explicit I/O contracts and can run independently or as part of integration tests. The design emphasizes data transformation across representations, including JSON puzzle structures, text knowledge bases with logical formulas, text solution and proof artifacts, and structured report dictionaries with text renderers.

Module Implementation Summary

Module 1, the Puzzle Generator (CSP), generates puzzle instances with entities, attributes, constraints, and a hidden solution. Its inputs are `grid_size` and `difficulty` (`easy`, `medium`, `hard`), and its output is a puzzle dictionary/JSON containing `entities`, `attributes`, `constraints`, `solution`, and metadata. Key design choices include solution-first generation, where a valid assignment is built first and then constraints are constructed to match it, along with a uniqueness enforcement loop in which candidate puzzles are validated using downstream logic (Module 2 plus bounded solution counting in Module 3) and regenerated if they are not uniquely solvable. Difficulty is scaled by a constraint-count multiplier. In an earlier version, we did not enforce uniqueness inside Module 1 and originally planned to check uniqueness later in the pipeline, but that approach created unstable regenerate/recheck cycles that could effectively become endless loops for some parameter combinations. We therefore moved uniqueness enforcement directly into Module 1 so non-unique candidates are rejected before being passed downstream. Module 2 consumes generated puzzle JSON directly, and the hidden solution is used later for verification and gameplay checking.

Module 2, Logic Representation (Propositional Logic), converts puzzle constraints into propositional formulas and assembles a structured knowledge base. Its input is the Module 1 puzzle structure, and its output is a text KB with facts, domain constraints, and puzzle constraints using standard connectives, including negation and biconditional patterns where needed. Key design choices include a canonical proposition symbol format (`E*_A*_V*`) for easy parsing, explicit domain rules that enforce at-least-one and at-most-one value constraints, and deterministic formula templates for each constraint type. Module 3 parses this KB for solving, while Modules 4 and 5 also consume it for checks and metrics.

Module 3, Puzzle Solving (Inference), solves the KB and outputs both a full assignment and a proof trace. Its input is Module 2 KB text, and its output is text containing solution lines, proof steps, and an inference-step count. The solving design applies forward-style constraint propagation first, then uses backtracking with deterministic branching and MRV-style variable selection if propagation is insufficient. It also includes a reusable bounded solution counter (`max_solutions` cap) used in uniqueness checks. Module 4 verifies the resulting assignment, Module 5 reads proof and inference signals, and Module 6 narrates proof steps.

Module 4, Solution Verification (Entailment and Constraint Checking), validates solver outputs against original constraints and KB-derived formulas. It takes Module 3 solution and proof text, Module 1 constraints with hidden solution, and Module 2 KB as inputs. It outputs a structured verification report (JSON-like dictionary) and a text rendering that includes overall pass/fail, per-constraint status, entailment checks, and violation summaries. Key design choices include a parse-and-check approach for both original constraints and parsed KB formulas, plus defensive handling for malformed or partial user solution text. Module 5 consumes verification outcomes as part of analysis context, and the UI uses verification for guess-check feedback.

Module 5, Complexity Analysis, computes explicit puzzle difficulty metrics and produces interpretable difficulty signals. It consumes the Module 1 puzzle structure, Module 2 KB text, Module 3 solution/proof text, Module 4 validation report text, and optionally a historical baseline dataset. It outputs both structured metrics and text reports that cover constraint count, search space size, inference step count, constraint density, logical formula complexity, branching factor, solution uniqueness, and overall difficulty score/label. Key design choices include named constants for threshold bands and fallback score components, a historical percentile scoring path with safe fallback when baseline data is missing or invalid, and label mapping to easy/medium/hard ranges. Module 6 uses this difficulty output as explanation context.

Module 6, Knowledge Representation and Solution Explanation, translates formal proof and solver outputs into human-readable reasoning. It takes Module 3 solution/proof, Module 2 KB, Module 1 constraints payload, and Module 5 difficulty report text as inputs. It outputs a structured explanation containing overall strategy, step-by-step reasoning, grouped reasoning phases (deductions, decisions, and other), and metadata summaries, along with text rendering for display or export. Key design choices include explicit parsing of proof and solution lines, phase grouping to improve readability, and template-based natural-language translation. One of our biggest Module 6 challenges was converting formal logical relationships into clue and explanation text that sounded natural to human players. Early versions were technically correct but read like machine output, with repetitive sentence starters, overly direct phrasing, and redundant clues that revealed the same relationship in multiple ways. We iterated by introducing mixed clue types (direct, negative, relational, either/or), pruning logically redundant clues, and adding wording variation so the output remained precise while feeling more like puzzle-book language. This highlighted a central knowledge-representation tradeoff: preserving formal correctness while still producing language users can understand quickly. The Streamlit UI can display explanation artifacts generated by this module.


Evaluation Methodology

Evaluation focused on correctness, robustness, and pipeline coherence through three complementary approaches: module-level unit tests, integration tests across adjacent and end-to-end module chains, and interactive UI checks for gameplay behavior. We evaluated generation validity and uniqueness behavior, formula translation consistency, solver correctness and proof signal extraction, verification accuracy for valid and corrupted attempts, complexity metric completeness and scoring behavior, explanation structure and readability, and UI gameplay loop correctness including hints, checking, and reveal behavior.

Primary metrics were automated test pass/fail counts, verification report fields such as `overall_pass`, violations, and entailment status, complexity metric availability and sanity checks, and Streamlit behavioral checks such as ensuring hidden solutions are not exposed in player mode and that correct guesses are revealed progressively. Test setup included random-seeded puzzle instances across difficulty levels and grid sizes, unit tests in `unit_tests/` aligned to module responsibilities, integration tests in `integration_tests/` for module-chain behavior, and bounded solution counting to avoid uncontrolled combinatorial cost during uniqueness checks.

Results were collected with `pytest`, and we report a representative aggregate from final development state: a full-suite pass at report time, with 124 tests passing in the local project state used during implementation. Additional focused test runs were used throughout iteration for touched modules and UI changes.

Results

Overall, the system met functional goals for all six modules and provided reliable end-to-end behavior under tested conditions. The strongest result is end-to-end correctness with modular transparency, where each stage remains independently inspectable while preserving compatibility with downstream parsers. Verification and reporting are also strong, since Module 4 provides clear per-constraint diagnostics and entailment-style checks that supported both testing and UI progress feedback. Difficulty analysis is explicit and interpretable because Module 5 reports all required metrics and supports both baseline-aware and fallback scoring modes, reducing brittleness when historical baselines are unavailable. Explainability exists as a first-class module through Module 6, which transforms formal proof traces into readable strategy and step narration rather than returning only solved/unsolved outcomes. Finally, the UI evolved toward puzzle gameplay by shifting from developer-output-heavy display to a game-first experience with natural-language clues, hidden answers, guess checking, progressive reveal, and completion celebration.

At the same time, several caveats remain. Clue quality is still template-driven, so stylistic naturalness and puzzle pedagogy vary by instance. Search scalability remains intentionally bounded to course-scale puzzle sizes and is not optimized for very large domains. Historical difficulty calibration also depends on dataset quality, so percentile-based interpretation is less meaningful without a strong baseline corpus.

Proposal Delta

This section summarizes what changed from the approved proposal and why. First, uniqueness enforcement in generation became stronger. The proposal required solvable puzzles and suggested validation, while implementation made uniqueness a hard constraint through bounded regenerate-until-unique behavior. This increased reliability for downstream solving and gameplay. Second, difficulty scoring was enhanced beyond static interpretation. The proposal required explicit difficulty metrics, and implementation added optional historical percentile scoring with fallback scoring so the system can generate a composite difficulty score/label for UI and explanation context. Third, UI scope evolved from pure module display toward interactive gameplay. Initial demos exposed internal artifacts, but we shifted default presentation to clue-first interaction with hidden answers and incremental reveal. Fourth, Module 6 output structure was refined. The proposal requested strategy and step-by-step explanation, and the final version added reasoning-phase grouping and metadata for clearer narrative flow. No required module was dropped or merged; scope changes centered on robustness and user-facing presentation quality.


Limitations and Failure Analysis

Natural-language clue generation remains vulnerable to repetitive or stylistically uneven output. A common failure case is that clue sets can feel formulaic when templates repeat similar relation structures. The likely cause is rule and template-driven verbalization without semantic variation control or discourse planning. Improvements would include adding a clue diversification layer with synonym templates and sentence planners, introducing clue-type quotas and redundancy penalties at generation time, and applying post-processing that enforces narrative cohesion and uniqueness.

Another limitation is that symbolic proposition assumptions constrain naming flexibility across solver and parser boundaries. A representative failure case appears when internal solving/parsing is attempted using fully real-world symbols, which can violate parser expectations built around canonical proposition forms. The likely cause is that Module 3 parsers were intentionally designed around canonical `E*_A*_V*` tokens produced by Module 2. Improvements include introducing a formal symbol-mapping abstraction between display labels and internal IDs, while preserving strict internal symbolic representations through reversible mapping tables.

A third limitation is that difficulty score interpretability is baseline-dependent. Historical percentile labels may drift when baseline data is sparse or unrepresentative, since relative scoring quality depends on distribution coverage. Improvements include building and versioning a curated historical benchmark set and reporting confidence intervals or baseline-coverage diagnostics.

Conclusions and Future Work

The project achieved its primary goal of delivering a coherent six-module AI system that moves from puzzle generation through formal reasoning, verification, complexity analysis, and human-readable explanation. The architecture demonstrates that symbolic pipelines can remain technically rigorous while still supporting user-facing interaction when module contracts and outputs are explicit and interpretable. Test results support functional stability for intended puzzle scales, and UI evolution demonstrates practical utility beyond static module demonstrations.

Future work should prioritize quality and scale in three areas. First, clue generation should move from template adequacy to puzzle-design quality, with better control over redundancy and pedagogical progression. Second, scoring should be grounded in stronger benchmark datasets so difficulty labels better reflect empirical player experience. Third, explanation quality could improve through richer discourse planning and optional visualization overlays showing why each constraint eliminates candidate assignments. A related next step is adding a dedicated language-realization layer that rewrites formal proof and clue structures into clearer, less repetitive puzzle-book English while preserving exact logical meaning.

In summary, the final system is complete, modular, and demonstrably useful as both an AI coursework artifact and a playable logic puzzle prototype.
